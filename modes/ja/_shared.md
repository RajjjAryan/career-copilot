### 退職ナラティブ（すべてのフレーミングで使用）

<!-- [カスタマイズ] あなた自身のナラティブに置き換えてください。例：
     - 「5年間自分のSaaSを構築・売却。今はエンタープライズAI応用にフルコミット。」
     - 「Series-Bで10倍成長中のエンジニアリングリード。次のチャレンジを求めて。」

`config/profile.yml` の退職ナラティブを使って、すべてのコンテンツをフレーミングする：

- **PDFサマリーで：** 過去から未来への橋渡し — 「同じ [スキル] を [JDのドメイン] に活かす。」
- **STARストーリーで：** `article-digest.md` のプルーフポイントを参照する。
- **ドラフト回答（ブロックG）で：** 転職ナラティブは最初の回答に盛り込む。
- **求人が「起業家精神」「オーナーシップ」「ビルダー」「エンドツーエンド」を求めている場合：** それが最大の差別化ポイント。マッチスコアの重みを上げる。

### 包括的な強み

プロフィールを **「実績のあるテクニカルビルダー」** としてフレーミングし、ロールに応じて調整：

- PM向け：「プロトタイプで不確実性を減らし、規律を持って本番投入するビルダー」
- FDE向け：「初日からオブザーバビリティとメトリクスで成果を出すビルダー」
- SA向け：「実際のインテグレーション経験でエンドツーエンドシステムを設計するビルダー」
- LLMOps向け：「クローズドループの品質システムでAIを本番に乗せるビルダー」

「ビルダー」をプロフェッショナルなシグナルとして位置づける — 趣味のレベルではない。実際のプルーフポイントが信頼性を裏付ける。

### 報酬インテリジェンス

<!-- [カスタマイズ] ターゲットロールの報酬レンジを調査し、値を調整してください -->

**一般的なガイドライン：**

- WebSearchで最新の市場データを取得（Glassdoor、Levels.fyi、OpenWork、転職会議、doda、マイナビ転職）
- スキルではなく職種名でフレーミング — 職種名が給与バンドを決める
- 日本の報酬は「年収」で表記が一般的（月給×12＋ボーナス）

### 日本市場の特殊性（重要）

日本の求人票や雇用契約には、EN/DE/FR市場に存在しない独自の概念がある。これらを正確に理解すること：

| 用語                              | 意味                                         | 評価への影響                                                              |
| --------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------- |
| **正社員**                        | 無期雇用の正規社員。社会保険・福利厚生完備   | 最も安定した雇用形態。DE の Festanstellung に相当                         |
| **契約社員**                      | 有期雇用契約（通常1年更新）                  | 安定性は低いが、報酬が高い場合あり。更新条件を確認                        |
| **業務委託 / フリーランス**       | 個人事業主として業務受託                     | 報酬は高いが社会保険なし。偽装請負リスクに注意                            |
| **派遣社員**                      | 派遣会社経由の就業                           | 直接雇用ではない。スキルアップ目的なら可だが長期は不向き                  |
| **年収**                          | 年間総報酬（基本給＋ボーナス＋手当）         | 報酬比較の基準。月給×何ヶ月かを確認                                       |
| **ボーナス（賞与）**              | 年2回が一般的（夏・冬）。業績連動の場合あり  | 「年収に含む」か「別途支給」か要確認。基本給×〇ヶ月で表記されることが多い |
| **みなし残業（固定残業代）**      | 一定時間分の残業代を月給に含む               | 超過分は別途支給が法的義務。含まれる時間数を必ず確認（20-45時間が一般的） |
| **試用期間**                      | 通常3-6ヶ月                                  | DE の Probezeit に相当。市場標準なのでリスクフラグではない                |
| **有給休暇**                      | 初年度10日、勤続年数で最大20日（法定）       | 企業独自の休暇制度（リフレッシュ休暇等）があればプラス                    |
| **退職金制度**                    | 長期勤続への報酬。確定拠出年金（DC）も増加中 | 長期雇用を前提とした福利厚生。外資系にはない場合も                        |
| **社会保険**                      | 健康保険、厚生年金、雇用保険、労災保険       | 正社員なら自動加入。業務委託の場合は自己負担                              |
| **ストックオプション（SO）/ RSU** | スタートアップ・外資系で一般的               | 上場前SOは流動性リスクあり。行使条件と税務を確認                          |
| **裁量労働制**                    | 実際の労働時間ではなく、みなし時間で賃金計算 | 長時間労働のリスク。ただしフレックスの自由度は高い                        |
| **フレックスタイム**              | コアタイム有無で柔軟な勤務時間               | リモートワークとの組み合わせを確認                                        |
| **転勤**                          | 会社都合による勤務地変更                     | 日系大企業では一般的。「転勤なし」は大きなプラス                          |

**みなし残業について確認する場合：**

**フォームの回答：**

- 「出社可能ですか？」の二択 → `profile.yml` の実際の勤務地に基づいて回答

### Time-to-Offer 優先

- 動くデモ＋メトリクス > 完璧さ
- 早く応募 > もっと学ぶ
- 80/20アプローチ、すべてにタイムボックスを設定

## グローバルルール

### 禁止事項

1. 経験や数値を捏造する
2. `cv.md` やポートフォリオファイルを変更する
3. 候補者に代わって応募を送信する
4. 生成メッセージに電話番号を含める
5. 市場水準以下の報酬を推奨する
6. 求人票を読まずにPDFを生成する
7. マーケティング用語や「ビジネスマナー定型文」を使う
8. トラッカーを無視する（評価した求人はすべて登録する）

### 必須事項

0. **カバーレター（送付状）：** フォームで送付状の添付が可能な場合、必ず添付する。CVと同じビジュアルデザインのPDFで作成。内容：求人票からの引用をプルーフポイントにマッピング、関連するケーススタディへのリンク。最大1ページ。
1. 求人を評価する前に `cv.md` と `article-digest.md`（存在する場合）を読む
   1b. **各セッション最初の評価時：** `node cv-sync-check.mjs` をBashで実行。警告があれば作業前に候補者に通知する
2. ロールのアーキタイプを検出し、フレーミングを調整する
3. マッチング時に履歴書の正確な行を引用する
4. WebSearchで報酬と企業データを調査する
5. 評価後にトラッカーに登録する
6. 求人票の言語でコンテンツを生成する（日本語の求人には日本語で）
7. 直接的かつ具体的に — 曖昧な表現は使わない
8. 生成テキストでは自然なテック日本語を使用。英語がテック業界で定着している用語はそのまま使う（Pipeline、Deploy、Sprint、OKR等）。無理に和訳しない
   8b. **PDFプロフェッショナルサマリーにケーススタディURL：** PDFがケーススタディやデモに言及する場合、URLは最初の段落（プロフェッショナルサマリー）に必ず含める。採用担当者はサマリーしか読まないことが多い。すべてのURLにHTMLで `white-space: nowrap` を設定
9. **トラッカー追加はTSV形式** — `applications.md` を直接新規行で編集しない。TSVを `batch/tracker-additions/` に書き出し、`merge-tracker.mjs` でマージする
10. **すべてのレポートヘッダーに `**URL:**` を含める**
