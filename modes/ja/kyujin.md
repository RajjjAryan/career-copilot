# モード：求人評価 — 完全評価 A-F

> **トリガー**：候補者が求人票（テキストまたはURL）を貼り付け、評価を依頼した場合。
> **出力**：スコア付きの構造化レポート。`reports/` に保存。

---

## 前提条件

評価の前に、以下のファイルをこの順番で読む：

```
view(path="modes/_shared.md")
view(path="modes/_profile.md")
view(path="cv.md")
view(path="article-digest.md")
```

---

候補者が求人票を貼り付けた場合（テキストまたはURL）、必ず全6ブロックを提供する。

## ステップ 0 — アーキタイプ検出

求人票を6つのアーキタイプのいずれかに分類する（`_shared.md` 参照）。ハイブリッドの場合は最も近い2つを示す。これにより以下が決まる：
- ブロックBでどのプルーフポイントを優先するか
- ブロックEでサマリーをどう書き換えるか
- ブロックFでどのSTARストーリーを準備するか

## ブロック A — ロール概要

以下のテーブルを作成：
- 検出されたアーキタイプ
- ドメイン（Platform / Agentic / LLMOps / ML / Enterprise）
- 機能（Build / Consult / Manage / Deploy）
- シニアリティ
- リモート（フルリモート / ハイブリッド / 出社）
- チーム規模（記載がある場合）
- TL;DR — 1文の要約

## ブロック B — 履歴書とのマッチ

`cv.md` を読む。求人票の各要件を履歴書の正確な行にマッピングするテーブルを作成する。

**アーキタイプ別の調整：**
- FDE → 迅速なデリバリーと顧客密着のプルーフポイントを優先
- SA → システム設計とインテグレーションを優先
- PM → プロダクトディスカバリーとメトリクスを優先
- LLMOps → Evals、オブザーバビリティ、パイプラインを優先
- Agentic → マルチエージェント、HITL、オーケストレーションを優先
- Transformation → チェンジマネジメント、導入推進、スケーリングを優先

**ギャップ（Gaps）** セクション — 各ギャップに対する軽減戦略：
1. ハードブロッカーか、あれば望ましい（Nice-to-have）か？
2. 隣接する経験で証明できるか？
3. ギャップをカバーするポートフォリオプロジェクトはあるか？
4. 具体的な軽減プラン（送付状での表現、短期のミニプロジェクト等）

## ブロック C — レベルと戦略

1. **求人票のレベル** vs **このアーキタイプにおける候補者の自然なレベル**
2. **「シニアとして売り込む」プラン**：具体的な表現、アーキタイプに合わせた強調ポイント、創業者経験をどう強みとして位置づけるか
3. **「レベルダウンされた場合」プラン**：報酬が適正なら受け入れる、6ヶ月後のレビューを交渉する、昇進基準を明確にする

## ブロック D — 報酬と需要

WebSearchで以下を調査：
- そのロールの現在の給与水準（Glassdoor、Levels.fyi、OpenWork、転職会議、doda、マイナビ転職）
- 企業の報酬に関する評判（OpenWork、Glassdoor）
- 日本市場におけるそのロールの需要トレンド

データとソースを記載したテーブルを作成。データが見つからない場合は正直に伝える — 捏造しない。

**日本市場 — 必須チェック項目：**
- 年収の内訳は？（基本給 × 何ヶ月 + ボーナス）
- ボーナス（賞与）の構成は？（固定 or 業績連動、年何回）
- みなし残業（固定残業代）は含まれているか？含まれている場合、何時間分か
- ストックオプション / RSU はあるか？
- 正社員か契約社員か業務委託か？雇用形態を確認
- 退職金制度はあるか？
- 福利厚生（社会保険、住宅手当、通勤手当、家族手当等）

## ブロック E — パーソナライゼーションプラン

| # | セクション | 現状 | 提案する変更 | 根拠 |
|---|-----------|------|------------|------|
| 1 | サマリー | ... | ... | ... |
| ... | ... | ... | ... | ... |

履歴書のトップ5変更 + LinkedInプロフィールのトップ5変更で、マッチを最大化する。

## ブロック F — 面接準備プラン

6-10のSTAR+Rストーリーを求人要件にマッピング（STAR + **振り返り（Reflection）**）：

| # | 求人要件 | STAR+Rストーリー | S | T | A | R | 振り返り |
|---|---------|-----------------|---|---|---|---|---------|

**振り返り（Reflection）** 列は、学んだことや今ならどうするかを記録する。これがシニアリティを示すシグナル — ジュニアは起きたことを説明するだけ、シニアはそこから教訓を引き出す。

**ストーリーバンク：** `interview-prep/story-bank.md` が存在する場合、ストーリーが既に登録されているか確認する。なければ新しいストーリーを追加する。時間の経過とともに、5-10のマスターストーリーの再利用可能なバンクが構築される。

**アーキタイプ別に選択・調整：**
- FDE → デリバリースピードと顧客密着を強調
- SA → アーキテクチャの意思決定を強調
- PM → ディスカバリーとトレードオフを強調
- LLMOps → メトリクス、Evals、本番環境のハードニングを強調
- Agentic → オーケストレーション、エラーハンドリング、HITLを強調
- Transformation → 導入推進と組織変革を強調

さらに以下も含める：
- 推奨するケーススタディ1件（どのプロジェクトをどう紹介するか）
- レッドフラグの質問と回答準備（例：「なぜ前職を辞めたのですか？」「チームの規模は？」「短期間での転職理由は？」）

## 評価後

ブロックA-Fの後に**必ず**実行：

### 1. レポート .md の保存

完全な評価を `reports/{###}-{company-slug}-{YYYY-MM-DD}.md` に保存する。

- `{###}` = 次の連番（3桁、ゼロ埋め）
- `{company-slug}` = 企業名を小文字、スペースなし（ハイフン使用）
- `{YYYY-MM-DD}` = 本日の日付

**レポート形式：**

```markdown
# 評価：{企業名} — {ロール名}

**日付：** {YYYY-MM-DD}
**アーキタイプ：** {検出結果}
**スコア：** {X/5}
**URL：** {求人票のURL}
**PDF：** {パスまたは未作成}

---

## A）ロール概要
（ブロックAの全内容）

## B）履歴書とのマッチ
（ブロックBの全内容）

## C）レベルと戦略
（ブロックCの全内容）

## D）報酬と需要
（ブロックDの全内容）

## E）パーソナライゼーションプラン
（ブロックEの全内容）

## F）面接準備プラン
（ブロックFの全内容）

## G）応募フォームのドラフト回答
（スコア >= 4.5 の場合のみ — 応募フォームの回答フィールド用ドラフト）

---

## 抽出キーワード
（ATS最適化のための求人票から15-20のキーワードリスト）
```

### 2. トラッカーに登録

`_shared.md` のグローバルルール（ルール9）に従い、トラッカーエントリは **TSV形式** で `batch/tracker-additions/` に書き出す。**`data/applications.md` を直接編集しない。**

エントリに含める情報：
- 次の連番（3桁ゼロ埋め）
- 本日の日付
- 企業名
- ロール名
- ステータス：`Evaluated`
- スコア：マッチ平均（X.X/5）
- PDF：❌（または📄、Auto-PipelineがPDFを生成した場合）
- レポート：レポートファイルへの相対リンク（例：`[001](reports/001-company-2026-01-01.md)`）

**TSVエントリ形式：**

```tsv
{num}\t{date}\t{company}\t{role}\tEvaluated\t{score}/5\t{pdf}\t[{num}](reports/{num}-{slug}-{date}.md)\t{note}
```

`data/applications.md` は最終的な統合トラッカーだが、更新は `merge-tracker.mjs` を通じたTSVベースの正規フローで行う。
